#### Challenge story — WebSocket dual-screen synchronization

```text
One of the key challenges was synchronizing teller and customer screens in real time during a banking transaction.

The teller and customer had to see consistent transaction state, but the backend still needed to be the source of truth. We had to handle WebSocket reconnects, customer confirmation, timeout, and message ordering carefully.

The approach was to create a sessionId and transactionId for each teller-customer interaction. Each WebSocket message carried session information, message type, sequence, timestamp, and correlationId. The backend validated every state transition and pushed only the required customer-facing information to the customer screen.

The result was a more reliable dual-screen flow and easier troubleshooting because each transaction could be traced through sessionId, transactionId, and correlationId.

Another challenging area was Kafka-based event handling for banking operations.

In banking, we cannot only think about publishing and consuming messages. We also need to think about event loss, duplicate processing, ordering, offset commit, and traceability.

The practical approach was to design consumers to be idempotent, commit offsets only after successful processing, use clear logging and correlation IDs, and monitor issues through Kibana. For business side effects, the system needed business keys or unique constraints to avoid duplicate results when a message was retried.

The main lesson was that in banking systems, at-least-once delivery plus idempotent processing is usually more realistic than claiming exactly-once end to end.
```

#### Challenge story — duplicate notification prevention

```text
One important challenge was avoiding duplicate notification processing.

Because the architecture is asynchronous and uses SQS, messages can be retried and delivered more than once. If the Lambda processes a duplicate message without checking state, the user may receive duplicate notifications.

To handle this, each event has an eventId or idempotency key. Before processing, the Lambda checks or writes a processing record in DynamoDB using a conditional write. If the event was already processed or is currently being processed, the duplicate can be skipped or handled safely. For failed processing, retry count and status are tracked, and after the maximum retry limit, the message can go to DLQ for investigation.

This design makes retries safe and helps the system handle high event volume without creating duplicate business side effects.
```

#### Challenge story — why SQS/EventBridge instead of synchronous API

```text
We used asynchronous event-driven architecture because notification processing should not block the main business flow.

EventBridge is useful as an event bus and routing layer. It allows events from different producers to be routed to different targets based on rules. SQS is useful as a queue because it buffers traffic spikes, decouples the producer from the consumer, supports retry, and provides DLQ for failed messages.

The trade-off is that the system becomes eventually consistent, and debugging can be harder. Because of that, we needed idempotency, structured logging, correlation IDs, CloudWatch metrics, and Slack alerts.
```

#### Challenge story — legacy migration correctness

```text
One major challenge was ensuring that the new Java batch jobs behaved consistently with the old VB6 and Lotus Notes system.

Legacy systems often do not have complete documentation, so we had to reverse engineer business rules from legacy behavior, existing code, sample data, and discussions with BA, QA, or business users.

The approach was to identify input and output for each job, document schedule and dependency, create golden datasets, compare old and new outputs, generate reconciliation reports, and get BA/QA sign-off before cutover.

The lesson was that legacy modernization is not only a rewrite. It requires understanding existing business behavior and proving that the new system produces correct results.
```

#### Challenge story — idempotent batch rerun

```text
Another important challenge was making batch jobs safe to rerun.

A batch job can fail after processing only part of the records. If we rerun the whole job blindly, it may create duplicate output or inconsistent status.

To handle this, the system should track job execution status, record-level status, and checkpoints. Each record should have a stable business key, and output operations should use upsert or unique constraints where possible. When rerunning a job for a business date, the processor can skip records that already succeeded and retry only failed or pending records, depending on business rules.

This makes recovery safer and reduces manual operation after failures.
```
